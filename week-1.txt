Software Requirements Specification (SRS)
1. Introduction
Purpose
The purpose of this system is to provide a secure and transparent platform for registering property details, validating ownership data, integrating external verification services, and storing a cryptographic hash on blockchain for tamper-proof records.
Scope
The system allows users to:
•	Register property details via a web interface
•	Validate user authentication using JWT
•	Check duplicate property entries
•	Generate SHA-256 hash of property data
•	Store data in MongoDB
•	Fetch and merge external verification data (DORIS, DLRC, CERSAI, MCA21)
•	Compute risk score
•	Store hash on blockchain (Ethereum smart contract)
•	Return confirmation with hash to user
2. Overall Description
Product Perspective
This is a web-based application with:
•	Frontend: React
•	Backend: Express (Node.js)
•	Database: MongoDB
•	Blockchain: Ethereum Smart Contract
•	External APIs: DORIS, DLRC, CERSAI, MCA21
System Architecture
The system follows a multi-tier architecture:
•	Presentation Layer (Frontend)
•	Application Layer (Backend APIs)
•	Data Layer (MongoDB)
•	Integration Layer (External APIs + Blockchain)
User Classes
General User: Registers property



3. Functional Requirements
1.	User Interaction
•	User shall open “Add Property” form
•	User shall input property details
•	User shall submit the form
2.	Authentication
•	System shall validate JWT token
•	If invalid → return HTTP 401
3.	Property Validation
•	System shall check if PropertyID already exists
•	If exists → return HTTP 409
4.	Hash Generation
•	System shall generate SHA-256 hash for property data
5.	Data Storage
System shall store property data in MongoDB
6.	External Data Integration
System shall call:
•	DORIS
•	DLRC
•	CERSAI
•	MCA21
Calls shall execute in parallel
7.	Data Aggregation
•	System shall merge external responses
•	System shall compute risk score
8.	Database Update
•	System shall update MongoDB with externalData
9.	Blockchain Integration
•	System shall invoke storeHashOnChain()
•	System shall store hash in Ethereum Smart Contract
10.	Blockchain Status Handling
•	If success → status = confirmed
•	If failure → status = failed
11.	Response Handling
•	System shall return HTTP 201 on success
•	System shall return generated hash to user
4. Non-Functional Requirements
1.	Performance
•	API response time < 3 seconds (excluding blockchain delay)
•	Parallel API calls for efficiency
2.	Security
•	JWT-based authentication
•	Data hashing using SHA-256
•	Secure API communication (HTTPS)
3.	Scalability
•	System shall support multiple concurrent users
•	Modular service-based architecture
4.	Reliability
•	System shall handle API failures gracefully
•	Retry mechanism for blockchain transactions (optional)
5.	Usability
•	Simple UI for property registration
•	Clear error messages
